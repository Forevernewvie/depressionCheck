# Safety Plan Feedback Hardening Prompt

다음 기준으로 `Safety Plan` 화면의 남은 프레젠테이션 책임을 정리하라.

## 목표
- `SafetyPlanScreen` 을 화면 조립 중심으로 단순화한다.
- 저장/전화/연락처 추가 후 사용자 피드백 정책을 화면 밖으로 분리한다.
- 연락처 헤더의 버튼 활성화, 개수 표시, 반응형 레이아웃 결정을 재사용 가능한 위젯/정책으로 분리한다.
- 사용자 가시 동작은 유지하면서 `SRP`, 테스트성, 유지보수성을 높인다.

## 필수 규칙
- 하드코딩 금지
- 중복 로직 금지
- 신규/수정 함수에 목적 설명 주석 추가
- `SnackBar` 메시지 정책은 전용 presenter/policy 에 모은다
- 화면은 `ScaffoldMessenger` 와 플랫폼 실패 메시지 조합을 직접 계산하지 않는다
- 관련 단위 테스트 또는 위젯 테스트를 추가한다

## 실행 절차
1. `savePlan`, `callPhone`, `addContact` 결과를 사용자 피드백 메시지로 변환하는 presenter 를 만든다.
2. 연락처 개수/추가 가능 여부/헤더 반응형 UI 를 재사용 가능한 정책 또는 위젯으로 분리한다.
3. `SafetyPlanScreen` 에 남아 있는 메시지 조합, 연락처 헤더 조건 분기, 반복되는 UI 피드백 호출을 제거한다.
4. 실패 경로에서 민감 정보가 노출되지 않는지 확인한다.
5. `flutter analyze` 와 관련 테스트 또는 전체 `flutter test` 를 실행한다.

## 검증 포인트
- 저장 성공/실패 메시지가 기존과 동일하게 동작하는가
- 전화 실패 시 사용자에게 일관된 피드백을 주는가
- 연락처 추가 성공 후 피드백이 유지되는가
- 연락처 추가 버튼 비활성화와 개수 표시가 한 곳에서 관리되는가
- UI layout 분기가 화면 본체가 아닌 전용 위젯/정책으로 옮겨졌는가
