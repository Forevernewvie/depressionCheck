# Architecture Quality Hardening Prompt

다음 기준으로 Flutter 앱 코드를 점검하고 직접 수정하라.

## 목표
- 확장성과 유지보수성을 최우선으로 구조를 개선한다.
- SOLID 원칙을 지키고, 객체지향 설계를 적용한다.
- 테스트 가능한 구조로 만든다.
- 보안, 예외 처리, 성능 병목 가능성을 사전에 분석한다.

## 필수 규칙
- 하드코딩 금지
- 매직넘버 금지
- 중복 로직 금지
- 환경변수와 런타임 설정 분리
- 의존성 분리 및 DI 적용
- 로깅 구조 포함
- 모든 신규/수정 함수에 목적 설명 주석 추가

## 실행 절차
1. 현재 코드에서 사용자 입력, 외부 API, 플랫폼 의존성, 저장소 접근, 네트워크 요청, 권한 요청 지점을 찾는다.
2. 하드코딩된 문자열, 숫자, URL, 에러 코드, 플랫폼 호출을 설정 객체나 전용 정책 클래스로 분리한다.
3. 직접 호출되는 외부 의존성을 인터페이스로 감싸고 Provider/DI로 주입한다.
4. 예외를 던지는 대신 typed failure 또는 result 모델로 반환 가능한지 검토한다.
5. 민감정보가 로그에 남지 않도록 로깅 컨텍스트를 검토한다.
6. 테스트가 어려운 부분은 fake/mock 가능한 형태로 추상화한다.
7. 최소 1개 이상의 단위 테스트를 추가해 성공/실패 경로를 모두 검증한다.
8. 변경 후 `flutter analyze` 와 관련 테스트 또는 전체 테스트를 실행한다.

## 산출물 형식
- 무엇을 왜 바꿨는지 요약
- 에러 가능 지점 분석
- 보안 고려 사항
- 성능 병목 가능성
- 남은 기술 부채와 다음 개선 우선순위

## 이번 실행 범위
- 외부 액션 계층(전화 연결, 지도 길찾기)부터 우선 적용한다.
- 이유:
  외부 URL/전화 호출은 보안, 플랫폼 의존성, 테스트성, 실패 처리 일관성이 동시에 필요한 고위험 영역이기 때문이다.
